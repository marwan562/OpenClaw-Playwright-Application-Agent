# OpenClaw plugin development

The repository follows the typed tool scaffold generated by OpenClaw `2026.7.1-2`. It uses `defineToolPlugin`, TypeBox input schemas, generated static metadata, and runtime validation. The installed simple-tool scaffold supports model tools but does not expose plugin-owned nested CLI registration; use the standalone `job-agent` CLI for direct terminal access.

## Build and validate

```bash
npm install
npm run plugin:build
npm run plugin:validate
npm test
```

`plugin:build` compiles TypeScript and regenerates `openclaw.plugin.json` from `dist/plugin/index.js`. Commit the generated manifest whenever tool metadata changes.

## Link into OpenClaw

These commands modify the user’s OpenClaw installation, so run them manually:

```bash
openclaw plugins install --link .
openclaw plugins enable job-agent
openclaw gateway restart
openclaw plugins inspect job-agent --runtime --json
```

Runtime inspection should list 18 tools, from `jobs_profile_import` through `jobs_emergency_stop`. If the Gateway uses a different working directory, configure the plugin `dataDir` with an absolute local path. Keep `dryRun: true` until a reviewed live adapter exists.

## Configuration

The generated plugin schema accepts only:

- `dataDir`: local SQLite and artifact directory.
- `dryRun`: defaults to `true`; prevents the service from proceeding to submission.
- `allowedSites`: defaults to `fixture,mock`; restrict this before enabling live adapters.

The plugin refuses profile import paths outside the linked project. It does not modify OpenClaw configuration, schedules, channels, or Gateway state itself.

## Tool behavior

Every input schema has `additionalProperties: false`. Tool handlers create the shared service, pass the OpenClaw cancellation signal, return schema-validated domain values, and close the database. The current OpenClaw simple-tool metadata format publishes input schemas; output strictness is enforced inside the service with Zod before the handler returns.

OpenClaw conversational scheduling should create an OpenClaw schedule that calls `jobs_campaign_run` with the stored campaign ID. The toolkit persists the cron and timezone as domain state; it does not silently edit the user’s OpenClaw scheduler.
